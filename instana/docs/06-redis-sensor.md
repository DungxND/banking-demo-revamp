# Redis Sensor

> **Source:** https://www.ibm.com/docs/en/instana-observability/current?topic=technologies-monitoring-redis
> Condensed for: Redis 8.x StatefulSet in `banking` namespace (no auth, no TLS)

---

## How It Works

The Instana Redis sensor is **automatically deployed** after the DaemonSet agent starts. It
auto-discovers the Redis pod via the containerd process scanner and activates a sensor for its
pod IP directly — no explicit host/port or NodePort needed.

```
DaemonSet agent pod (instana-agent namespace)
  └─ auto-discovers redis container via containerd
       └─ TCP connect → <redis-pod-ip>:6379 (every 10 s)
            └─ INFO, CONFIG GET, SLOWLOG GET, LATENCY LATEST
```

> The agent log shows: `"Activated Sensor for <pod-ip>:6379"` for the auto-discovered entity.
> Do **not** add an explicit `hosts:` block — it creates a duplicate Redis node on the
> dependency graph that the agent cannot merge with the auto-discovered entry.

## Requirements on Redis Side

- The `CONFIG` command must **not** be disabled or renamed
- The `INFO` command must be accessible
- No password required if Redis runs without auth (banking-demo default)

## Agent Configuration

From [`instana/configuration.yaml`](../configuration.yaml):

```yaml
com.instana.plugin.redis:
  use_ssl: false
  # hosts: []  ← intentionally omitted; auto-discovery handles Redis
```

`use_ssl: false` suppresses the transient `"SSL is disabled… Read timed out"` warning at
sensor init. No `hosts:` block is needed — the DaemonSet agent auto-discovers the Redis pod
from the containerd process list and connects to its pod IP directly.

### ACL Permissions (Redis 6+ — not needed for banking-demo)

If ACL is enabled, create a monitoring user with minimum permissions:

```redis
# In redis.conf or via redis-cli
ACL SETUSER instana-monitor on >password123 ~* -@all \
  +info +config|get +slowlog|get +pubsub|channels +pubsub|numpat +latency|latest
```

## How banking-demo Uses Redis

banking-demo services use Redis for:

- **Session storage** (`auth-service`) — `session:{sid}` keys, TTL-based expiry
- **User cache** (`auth-service`) — `user_cache:{username}` for fast login lookup
- **Balance read model** (`account-service`) — `balance:{user_id}` updated on every transfer event
- **Pub/sub notifications** (`notification-service`) — `transfer_notify` channel for real-time WebSocket push

All Redis access is from Go services using the [`go-redis`](https://github.com/redis/go-redis) client
via [`internal/redis`](../../internal/redis/redis.go).

## Client-Side Tracing (instaredis — active)

Per-command Redis spans are generated by `instaredis` via the Instana go-sensor. Every Redis
command executed by a consumer service appears as a child span in the Instana trace waterfall,
linked to the parent NATS consumer span.

Wired in [`internal/redis/redis.go`](../../internal/redis/redis.go):

```go
import instaredis "github.com/instana/go-sensor/instrumentation/instaredis/v2"

if c := tracing.Collector(); c != nil {
    instaredis.WrapClient(client, c)
}
```

Result in the Instana UI: `auth-service` → NATS span → `GET session:{sid}` Redis span.

> **OTel Redis spans** (via `go-redis/extra/redisotel`) are not used — `instaredis` is the
> active path. Both would produce duplicate spans if enabled together.

## Verifying in Instana UI

1. **Infrastructure → Kubernetes → `banking`** — Redis entity with memory, ops/sec, client count
2. **Applications → Dependencies** — Redis node connected to auth/account/notification-service arcs
3. **Analytics → Calls** — filter `db.type=redis` to see all instaredis spans

```bash
# Check agent auto-discovered Redis
kubectl -n instana-agent logs ds/instana-agent --tail=50 | grep -i "redis"
# Expected: "Activated Sensor for <pod-ip>:6379"

# Test Redis connectivity from inside a pod
kubectl -n banking exec deploy/auth-service -- \
  sh -c 'redis-cli -u $REDIS_URL ping'
# Expected: PONG
```

Common issues:
- Two Redis nodes on the dependency graph — an explicit `hosts:` block was added alongside auto-discovery; remove it from `configuration.yaml`
- `redis_config_command_unavailable` — `CONFIG` was renamed; set `config-command` in `configuration.yaml`
- `SSL is disabled… Read timed out` — transient at startup; resolved by `use_ssl: false`
