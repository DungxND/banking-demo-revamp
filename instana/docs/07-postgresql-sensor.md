# PostgreSQL Sensor

> **Source:** https://www.ibm.com/docs/en/instana-observability/current?topic=technologies-monitoring-postgresql
> Condensed for: PostgreSQL 18 StatefulSet in `banking` namespace

---

## How It Works

The Instana PostgreSQL sensor is **automatically deployed** after the DaemonSet agent starts.
It connects to PostgreSQL using the credentials in `configuration.yaml` and queries `pg_stat_*` views.

```
DaemonSet agent pod (instana-agent namespace)
  └─ JDBC connect → postgres.banking.svc.cluster.local:5432
       └─ SELECT from pg_stat_activity, pg_stat_user_tables,
                      pg_stat_bgwriter, pg_locks, pg_database
```

The DaemonSet agent reaches `postgres.banking.svc.cluster.local` via cluster-internal DNS —
no NodePort is needed.

## Required: Enable Statistics Collection

The sensor needs PostgreSQL statistics tracking enabled. This is done in banking-demo via the
init ConfigMap in [`helm/templates/postgres-init-configmap.yaml`](../../helm/templates/postgres-init-configmap.yaml).

### What's required in `postgresql.conf`

```sql
track_activities = on    -- monitors current command per connection
track_counts = on        -- cumulative stats for table/index access
track_io_timing = on     -- block read/write times
```

### Persistent config (survives restarts)

```sql
ALTER SYSTEM SET track_activities = 'on';
ALTER SYSTEM SET track_counts = 'on';
ALTER SYSTEM SET track_io_timing = 'on';
SELECT pg_reload_conf();

-- Verify
SHOW track_activities;
SHOW track_counts;
SHOW track_io_timing;
```

## Agent Configuration

From [`instana/configuration.yaml`](../configuration.yaml):

```yaml
com.instana.plugin.postgresql:
  user: 'banking'
  password: 'bankingpass'
  database: 'banking'
  poll_rate: 10
```

Schema source: [IBM docs — Monitoring PostgreSQL](https://www.ibm.com/docs/en/instana-observability/current?topic=technologies-monitoring-postgresql)

> The `database` field is used for authentication only. Leave it empty to default to the value of `user`.

> **Do NOT add `password` or `key` to `com.instana.secrets.list`.**
> Instana applies the secrets filter to its own sensor config fields before connecting.
> `password` in the list redacts the sensor password to `""` → SCRAM failure → `08004`.
> `key` is too broad and redacts `INSTANA_AGENT_KEY`.

> **Password must match the cluster Secret** (`helm/values.yaml` `secret.postgresPassword`).

## Metrics Collected

| Category | Metrics |
|----------|---------|
| Connections | `numbackends`, `max_conn`, active/idle/waiting |
| Transactions | `xact_commit`, `xact_rollback`, TPS |
| I/O | `blks_read`, `blks_hit`, cache hit ratio |
| Locks | Lock types, wait count |
| Tables | Rows inserted/updated/deleted, seq/idx scans |
| Background writer | `buffers_clean`, `checkpoints_timed` |
| Replication | LSN lag (if replicas present) |
| Query performance | Top slow queries (if `pg_stat_statements` enabled) |

## How banking-demo Uses PostgreSQL

All banking-demo services connect to PostgreSQL via the `pgxpool` connection pool (Go, via
[`internal/db`](../../internal/db/db.go)). The pool is initialised once at startup with the
`DATABASE_URL` environment variable.

Key tables:

| Table | Owner service | Purpose |
|-------|--------------|---------|
| `users` | auth-service | User accounts and hashed passwords |
| `accounts` | account-service | Account balances (source of truth) |
| `transfers` | transfer-service | Transfer records (uses `SELECT FOR UPDATE`) |
| `notifications` | notification-service | Notification log per user |

## Client-Side Tracing (instapgx — active)

Per-query PostgreSQL spans are generated by `instapgx` via the Instana go-sensor. Every SQL
query executed by a consumer service appears as a child span in the Instana trace waterfall,
linked to the parent NATS consumer span.

Wired in [`internal/db/db.go`](../../internal/db/db.go):

```go
import instapgx "github.com/instana/go-sensor/instrumentation/instapgx/v2"

if c := tracing.Collector(); c != nil {
    cfg.ConnConfig.Tracer = instapgx.InstanaTracer(cfg.ConnConfig, c)
}
```

Result in the Instana UI: `auth-service` → NATS span → `SELECT * FROM users WHERE ...` PostgreSQL span.

> **OTel PostgreSQL spans** (via `otelpgx`) are not used — `instapgx` is the active path.
> Both would produce duplicate spans if enabled together.

## Verifying in Instana UI

1. **Infrastructure → Kubernetes → `banking`** — PostgreSQL entity with connections, TPS, cache hit ratio
2. **Applications → Dependencies** — PostgreSQL node connected to auth/account/transfer/notification arcs
3. **Analytics → Calls** — filter `db.type=postgresql` to see all instapgx spans

```bash
# Check agent sensor is active
kubectl -n instana-agent logs ds/instana-agent --tail=50 | grep -i "postgresql\|postgres"
# Expected: "Activated Sensor ... postgresql"

# Verify DB connectivity from inside a pod
kubectl -n banking exec deploy/auth-service -- \
  sh -c 'psql $DATABASE_URL -c "SELECT version();" 2>&1 | head -1'
# Expected: PostgreSQL 18.x ...

# Check stats are enabled
kubectl -n banking exec deploy/auth-service -- \
  sh -c 'psql $DATABASE_URL -c "SHOW track_counts;"'
# Expected: on
```

Common issues:
- `postgresql_connection_failed` — check credentials in `configuration.yaml` match `helm/values.yaml` `secret.postgresPassword`
- Stats views returning 0 — `track_counts`/`track_io_timing` not enabled; run `ALTER SYSTEM SET ...` above
- `Postgres password is empty string` error every 10 min — agent auto-discovered the pod and started a second credential-less sensor; this is benign and does not affect the configured sensor
